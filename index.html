Okhttp源码阅读笔记(1)

Motivation/Target

addInterceptor与addNetworkInterceptor区别

enqueue异步发送网络请求

首先说明okhttp中发送网络请求的流程:

  • OkHttpClient::newCall(Request)来创建一个Call的对象, 其中Request对象也是通过建造者模式来构造

  • 我们在使用的时候通常使用Call 的 execute(同步) 或者 enqueue(异步)来通过传入callback的方式来让其执行callback中的逻辑, 下面我们以enqueue 为例来分析callback的调用时机

  • 首先需要指明一点: 在OkHttpClient中Call的具体实现类为RealCall

  • 在enqueue中首先会检查是否被执行过(只能执行一次), 其次再调用transmitter.callStart(), 最后调用client中的dispatcher来把callback包装进AsyncCall后压入dispatcher中的队列
  • Dispatcher会维护两个队列readyAsyncCalls和runningAsyncCalls, 在client.newCall情况下forWebSocket为false故会首先检查两个队列中是否有和当前call host相同的, 有的话就使用同一个AtomicInterger变量来计数当前对目标host的请求数量
  • 不太清楚为什么在进入方法的时候必须assert当前线程(应该是主线程)没有拿着dispatcher的锁
  • 这个方法就是挑出了能运行的asyncCall然后去让他们在线程池上去执行
  • 而这个asyncCall的真正逻辑是在其execute方法里去执行的, 交由线程池执行的核心代码也是在这个函数中
  • 如果asyncCall被线程池拒接了, 那么会在主线程上调用callBack.onFailure

在这个函数里可以看到, 真正的response是由getResponseWithInterceptorChain()返回的, 然后把这个response交由callback.onResponse去处理

可以看到在非forWebSocket的情况下, interceptor的顺序为:

  • 通过addInterceptor添加的interceptors
  • RetryAndFollowUpInterceptor
  • BridgeInterceptor(如果cookieJar不设置的话为NO_COOKIES为空)
  • CacheInterceptor
  • ConnectInterceptor
  • 通过addNetworkInterceptor添加的interceptors
  • CallServerInterceptor: 最后一个添加的interceptor, 也是实际向服务器发送网络请求的

然后再根据上面的intercetpors list来构建RealInterceptorChain对象来开始递归的proceed, 最后返回Response对象

自带的Interceptors

RetryAndFollowUpInterceptor
  • 注释: This interceptor recovers from failures and follows redirects as necessary. It may throw an IOException if the call was canceled.
  • 和他的名字一样, 就是去做retry和followup的, 配置为最多跟redirect20次, 超过的话会抛Exception
BridgeInterceptor
  • 注释: Bridges from application code to network code. First it builds a network request from a user request. Then it proceeds to call the network. Finally it builds a user response from the network response.
  • 在发送请求之前根据RequestBody来填充标准http报文需要的Content-Type, Connection, Content-Length等参数, 设置cookies
  • 在收到网络发回的response之后会strip掉一些没什么用的header, 把cookies填充到cookieJar里面, 解压body content(只限于gzip格式)
CacheInterceptor
  • 注释: Serves requests from the cache and writes responses to the cache.
ConnectInterceptor
  • 注释: Opens a connection to the target server and proceeds to the next interceptor.
CallServerInterceptor
  • 注释: This is the last interceptor in the chain. It makes a network call to the server.

Result

我们根据通过两个不同的方法添加的interceptors在自带interceptors中的顺序就可以看出:

  • 通过后者(addNetworkInterceptor)添加的拦截器只在真正向网络发送请求的时候才会起作用(如果是从cache里取就不会被调用了), 而且接触到的是raw Request和raw Response, 包含了http报文的全部信息
  • 通过前者(addInterceptor)添加的拦截器会接触到每一次app发送的request和收到的response, 而中间通过了多少redirect之类的并不是其关心的内容
  • 应该在NetworkInterceptor里做依赖于网络状态而与返回内容无关的操作, 在Interceptor去做和返回内容有关的操作

Dynamic Proxy

动态代理指通过调用Proxy类的

来动态的创建代理的方法, 通过该方法创建的代理实现了interfaces中的接口, 当通过返回的代理调用方法时, 会转发调用至h: InvocationHandler上, 通过h来实现最终的调用

对interfaces的一些限制:

  • 其中的元素必须是接口
  • 不能有重复的元素
  • 所有的接口和内部的public方法必须在传入的classloader下通过名字可见

如果interfaces中元素顺序调换会得到两个不同的proxy实例

Compose! (1)

知之为知之, 不知为不知, 是知也

纸上得来终觉浅, 绝知此事要躬行

compose的alpha版出来也有一段时间了, 最近想写一个安卓项目练练手(老不写脑子生锈了都), 既然compose出了, 那自然就不再写那堆恶心的xml了! 学习compose走起

Intro

Compose的设计逻辑和原生安卓View那一套是完全不同的, 在原生中我们一般使用findViewById()来获取我们需要的View的对象, 每个View是会维护一个internal state的, 我们可以说是“面向View对象”的编程

但在compose中, view是relative-stateless的, 他通过observe State类的变量来达到更新UI的目的, 这一步在compose中叫做recompose, 当事件发生状态改变时会尽可能少的全部重新绘制相关组件, 这种思想其实是非常适合与ViewModel进行协作的. 我们可以在widget的监听事件中更新ViewModel中的livedata, UI本身使用拓展函数LiveData::observeAsState得到的State来绘制自己, 这样在event发生后相关的widget会recompose

如上图所示, 这就是unidirectional pattern

Example

我们考虑一个简单的仅包含一个在button上显示其被点击次数的activity

我们考虑一个使用ViewModel的情况(在这个情形下是不必要使用viewModel的, 因为对数据的更新非阻塞直接在主线程进行, 传统方式直接在MainActivity里面维护一个count:Int就可以, compose方式在compose fun里面自己实现一个MutableState代理count变量就可以)

在传统方式中,

我们必须持有要更新的UI widget的对象, 并显式更新他们

在compose中,

可以看到我们的数据和UI是完全分离的, 我们把数据包装成了compose-observable的State对象, 在liveData中数据更新时, 所有observe它的compose fun都会自动的recompose, 我们无须再去显式的更新所有依赖于此数据的widget.

Scope fun

Context object
  • run, with, apply的context refer为this, 因为其参数为T.() -> R 形式, 所以能直接用this做引用
  • let, also 的context refer为it, 其参数为(T) -> R, 只不过在函数内部调用了block(this)
Return value
  • apply and also return the context object.
  • let, run, and with return the lambda result.
Facts

  • T.run{ } 和 with(T){ }等价

  • let相比于run和with区别在于它的参数block是(T)->R中可以执行T外部语句

  • apply和also区别也在于also的参数block可以执行T外部语句

  • 这个是一个experimental的功能, 用于告知编译器block只会被执行一次, 如果不加这个contract, block内部不可以对外部的val变量赋值

State Hoisting

caller持有callee需要的状态, 并且提供给callee其需要的更新状态的接口

state hoisting这一设计模式让我们可以写出纯stateless的composable, 并且把他们的状态hoist到上层

remember(){}会在compose内部保存状态, remember会在()内填入的参数不变的情况下不在recompose的时候重新执行{}这一block

72.Edit Distance

Posted on 2020-11-07 In leetcode

72.Edit Distance

source link

Description

Given two strings word1 and word2, return the minimum number of operations required to convert word1 to word2.

You have the following three operations permitted on a word:

  • Insert a character
  • Delete a character
  • Replace a character
Signature
Analysis

dp[i][j] represents minDistance(word1[:i], word2[:j])(inclusive)

if word1[i]==word2[j] => dp[i][j] = dp[i-1][j-1]

else dp[i][j] = min(dp[i-1][j]+1, dp[i][j-1]+1, dp[i-1][j-1]+1)

核心思想还是创建一个二维的dp数组, 只要找到递推表达式就简单了, 其中word1[i]!=word2[j]对应的三种情况就是:

  • dp[i-1][j]+1: word1往前退一个, 然后把word1[i]删掉(所以加1)
  • dp[i][j-1]+1: word2往前退一个, 然后把word2[j]插入进来
  • dp[i-1][j-1]+1: word1, word2都往前退一个, 然后把word[i]替换为word[j]
Solution

32.Longest Valid Parentheses

Posted on 2020-11-04 In leetcode

32.Longest Valid Parentheses

source link

Description

Given a string containing just the characters '(' and ')', find the length of the longest valid (well-formed) parentheses substring.

Signature
Analysis
  1. 在对数组进行操作的时候要时刻注意越界问题, 特别在循环中必须保证在access数组时, 在界内
  2. 在创建栈的时候一开始脑子抽了感觉能把长度设置为n/2+1省点空间… 被“(((((()”这个testcase教做人

接下来就是main idea了:

  1. 首先我们创建一个与字符串等长且初始为0的数组, 这个数组第i个元素只有在字符串这一位是’)’时可能非0, 代表了从这个’)’往前看最长连续的有效括号长度
  2. 在遍历的时候, 每当遇到’(’, 我们就把他的index压入栈中
  3. 如果遇到了’)’, 这意味着要consume一个’(’, 我们检查栈中是否为空, 如果是空的话就什么都不做(毕竟初始已经把长度设置为0了)
  4. 如果此时栈非空就consume一个’(’, 检查他前一位是否非0, 如果是的话就加上’(’前一位在array中的值
  5. 再加上这个’)’前一位在array中的值, 这对应了嵌套的情况, 即连续出现多个’)’
  6. 上面的步骤很大程度上都依赖了array初始值都为0, 这样就可以不用保留其他信息来随便加(如果是0的话无影响)
  7. 最后把array中最大元素返回就可了

感觉dp就是要充分利用之前的信息, 其实好像所有算法的目的都是要“customized”, 为特定的问题使用尽可能少的信息, 排除尽可能多的冗余信息来给出解
