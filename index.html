493.Reverse Pairs

Posted on 2020-10-15 In leetcode , Binary Index Tree

493.Reverse Pairs

source link

Description

Given an array nums, we call (i, j) an *important reverse pair* if i < j and nums[i] > 2*nums[j].

You need to return the number of important reverse pairs in the given array.

Signature
Analysis

这个题本质上和315.Count of Smaller Numbers After Self是一样的, 都是对数组中每个元素求后面元素满足xx条件的个数再求和, 并且一定得保证nlogn的复杂度

  • Approach 1: binary search tree

    以红黑树为例, 在node_t结构中加上左(右)子树中元素总数, 对nums从前往后进行遍历, 每轮遍历查询红黑树中严格比自己二倍大的元素个数再插入自身到红黑树中

    由于红黑树中查找插入包括查找成功后向上回溯查找index都是logn的操作, 三个logn操作总体上是nlogn的复杂度

  • Approach 2: binary indexed tree

    使用bit的话相当于对一个和nums等长但是每一位标志着sort后nums在该位index是否被遍历到了的(遍历到是1, 没遍历到是0)的数组维护一个bit, 先创建一个copy作nums备份, 然后sort copy, 再遍历nums, 每轮查找自身二倍元素在copy中的index然后求这个index往后的区域和(在原数组中index比他小的元素有多少大于它的二倍), 然后把自身在copy中的index插入bit中

    对数组排序, bit中插入查询以及在有序数组中的查找复杂度都是logn, 所以总体上复杂度是nlogn

  • Approach 3: merge sort

    对每次mergesort, 在合并两个subarray之前, 遍历后一个subarray, 对每个元素自身的二倍在前一个有序的subarray中作二分查找, 这样能得到这次做mergesort的subarray中Reverse Pairs的总数(其实也是divide and conquer思想, 在这位老哥总结的general principles中算是Partition recurrence relation, 二分的拆分成子问题)

    merge sort总的要进行logn次, 每次merge sort中查找是logn的复杂度, 但合并是n的复杂度, 故整体是nlogn的复杂度

这三种方法第一个是bst过不了测例, 红黑树太麻烦了; 第二个一直超时也不知道为啥; 最后用merge sort写的总算过了
